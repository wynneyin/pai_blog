## 前言

这份整理覆盖了腾讯大模型应用开发（Agent 方向）一面的核心考察点，共 15 个问题。不是背诵题，而是实际工程中会遇到的真实问题，建议结合自己的项目经验理解记忆。

---

## Q1. Skill 上下文膨胀问题

**把所有 Skill 塞进 System Prompt 的弊端**

- 上下文窗口被占满，压缩有效对话空间
- 候选技能噪声过大，干扰模型判断
- 模型在相似技能间更容易混淆，导致误调用

**核心解决方案：Skill 外部化 + 动态注入**

System Prompt 只保留最小调用规则和协议，Skill 描述做成外部注册表，按需动态加载。

**工程实现三步走**

1. **增加 Skill 路由层**：通过规则引擎、轻量分类模型或向量检索，先筛选出与当前任务最相关的 Top-K 个 Skill
2. 仅将 Top-K Skill 的描述、参数 Schema 和 Few-Shot 示例拼接进 Prompt
3. **长 Skill 分层处理**：拆分为摘要版和完整版，先注入摘要供模型选择，确认调用后再加载完整说明

## Q2. 相似 Skill 误调用问题

**核心思路**

不要堆砌模糊的自然语言描述，而是给 Skill 增加结构化判别信息，从定义、选择、校验全流程降低误召率。

**四步解决法**

1. **Skill 结构化定义升级**：从纯文本描述扩展为包含适用场景、禁用场景、输入参数、返回格式、前置条件、失败条件和调用示例的完整结构
2. **两阶段选择机制**：先粗召回 Top-K 候选，再让模型根据参数约束和正反样例做精排（Rerank）
3. **负样本增强**：明确告诉模型"什么问题绝对不应该调用这个 Skill"，补充边界案例
4. **执行前强校验**：增加 Schema 校验和业务规则检查，即使模型选错也能拦截误调

**长期根治方案**

如果两个 Skill 语义长期接近且持续混淆，应在系统设计层面合并为一个 Skill，内部通过参数路由区分不同逻辑。

## Q3 & Q4. Agent 设计的核心：上下文工程

**核心认知**

Agent 不是单轮问答系统，而是持续决策系统。其稳定性完全取决于：模型能看到什么信息、信息的顺序和优先级、以及如何区分规则、证据和历史状态。

> Agent 设计的核心不是「给模型更多信息」，而是**「给模型正确的信息」**。

**关键注意点**

- **严格角色隔离**：明确 System、Developer、User、Tool Output、Memory 的优先级和边界，绝对不能将外部文本和系统规则混在一起
- **精准信息裁剪**：只保留与当前任务直接相关的内容，避免将所有历史对话、检索结果和工具说明一股脑灌入
- **强制结构化格式**：工具返回结果统一用 JSON 格式，任务状态单独字段化管理，长历史对话做摘要而非堆砌原文
- **可信度分层机制**：系统规则 > 用户输入；工具执行结果 > 模型猜测；外部知识证据 > 无依据生成
- **长任务上下文治理**：实现上下文压缩和过期淘汰机制，防止状态漂移和旧信息污染

## Q5. Agent 长期记忆系统设计

**核心认知**

长期记忆 ≠ 存储所有历史对话。一个可用的长期记忆系统必须回答三个问题：什么值得写回、什么时候应该遗忘、冲突信息如何处理。

**写回策略**

- 只存储高价值稳定信息：用户偏好、身份属性、长期任务目标、反复出现的业务事实
- 结构化抽取存储：将自然语言信息转化为结构化字段（如将「用户更喜欢用 Python」存入 `profile.language` 字段）

**衰减策略**

- 记忆带有时效性和权重属性
- 短期热点信息赋予高权重，随时间推移自动衰减
- 长期稳定偏好低频更新，不轻易删除

**冲突消解策略**

结合时间戳、来源可信度和置信度三个维度判断：新的高可信事实覆盖旧事实，低可信内容仅作为候选，不直接替换已有信息。

**进阶架构：事件流 + 物化视图**

所有记忆写入先以事件（Event）形式持久化，再异步聚合成用户画像（Profile）等物化视图，既方便审计，也支持任意时间点的状态回滚。

## Q6. 并行 Tool Calling 工程实践

**核心认知**

并行 Tool Calling 的核心不是「同时调用多个接口」，而是解决**依赖关系、状态一致性、结果合并、失败回滚**四大问题。

| 问题 | 解法 |
|------|------|
| 依赖管理 | 只有完全互不依赖的工具才能并行；有依赖则串行 |
| 一致性保证 | 临时观察区 + 统一合并，避免并发写污染全局状态 |
| 可回放能力 | 每次调用携带 trace_id、step_id、输入输出，组成完整 DAG |
| 失败处理 | 按工具粒度重试，不因单个超时丢弃所有结果 |
| 副作用防护 | 发消息、创建订单等操作必须添加幂等键，防止重复写入 |

## Q7. 主流 Agent 架构设计模式

**单 Agent 模式**

| 模式 | 说明 |
|------|------|
| **ReAct** | 思考 → 调用工具 → 观察结果 → 继续思考，最经典的模式 |
| **Plan-and-Execute** | 先制定整体规划，再按步骤执行，适合长周期复杂任务 |
| **Router** | 先对任务分类，再路由给专门的子 Agent 或工具链 |
| **Workflow+Agent** | 固定业务流程中嵌入局部自主决策点，线上生产使用最多 |

**多 Agent 模式**

| 模式 | 说明 |
|------|------|
| **Supervisor** | 主 Agent 负责任务拆解和分派，多个子 Agent 并行执行 |
| **Blackboard** | 多个 Agent 通过共享的全局状态黑板进行协作 |
| **点对点自由协作** | Agent 之间直接通信，治理难度极高，线上极少使用 |

## Q8. MCP、A2A 与 Function Calling 的本质区别

| 技术 | 本质定位 | 核心解决问题 |
|------|----------|------------|
| **Function Calling** | 单机式工具调用协议 | 模型如何调用本地或受控工具 |
| **MCP** | 标准化的上下文和能力暴露协议 | 外部能力如何以标准方式被模型客户端发现和调用 |
| **A2A** | 智能体间协作协议 | 多个智能体如何分工完成复杂任务 |

**三者抽象层级**

- Function Calling：调用一个函数
- MCP：暴露一组标准能力
- A2A：多个智能体协作完成任务

**工程选型指南**

- 单应用内简单工具调用 → Function Calling
- 多模型客户端/IDE 共享一套工具和资源 → MCP
- 多 Agent 分工协作、任务转派和结果汇总 → A2A

## Q9. RAG 接入 Agent 的正确设计方式

**核心认知**

RAG 接入 Agent 后，不应该是固定的「先查再答」流程，而应该作为一个**自主决策节点**，由模型自主判断是否需要检索、何时检索。

**标准工程实现**

1. **将 RAG 封装为独立工具**：由 Agent 自主决定何时调用、调用多少次
2. **检索结果预处理**：去重过滤 → 语义切片 → 相关性排序 → 来源可信度标注
3. **幻觉抑制机制**：强制要求模型只根据检索到的证据回答，无证据时明确说明「不确定」

## Q10. Tool Calling 工程实现

**与普通函数调用的本质区别**

- **普通函数调用**：调用逻辑写死，调用者、函数名、参数全由开发者控制
- **Tool Calling**：模型决定调用意图，宿主系统决定能否执行，必须增加校验和调度层

**标准四层工程架构**

```
工具注册层   → 统一管理所有可用工具的元信息
参数 Schema 层 → 定义每个工具的输入输出格式和约束
执行器层     → 负责实际调用工具并处理返回结果
审计日志层   → 记录所有工具调用的全链路信息
```

**上线安全红线**

- 服务端必须做：白名单校验、参数合法性校验、超时控制、幂等控制、错误包装
- **绝对不能把工具执行权直接交给模型**

## Q11. Agent Memory 系统通用设计

**三层记忆架构**

| 层级 | 内容 | 作用 |
|------|------|------|
| 短期记忆 | 最近 N 轮对话原文 + 当前任务状态 | 保证会话连贯性 |
| 长期记忆 | 用户偏好、身份属性、长期任务目标 | 跨会话的稳定知识 |
| 摘要记忆 | 久远历史对话的关键事实提炼 | 大幅压缩 Token 占用 |

**落地最佳实践**

采用混合存储 + 按需召回策略：
- 保留最近 N 轮对话原文
- 更久远的内容压缩为结构化摘要
- 关键信息（如用户手机号、邮箱）单独抽取为字段存储
- 生成时根据当前任务语义召回相关记忆，而非无脑全量拼接

## Q12. Prompt Injection 防护

**Agent 场景的特殊风险**

在 Agent 场景下，Prompt Injection 不仅会生成错误文本，还可能导致**工具误调用、越权执行、触发真实业务操作**，风险远高于普通对话场景。

**五层防护体系**

1. **角色和优先级隔离**：系统规则优先级最高，外部知识绝对不能覆盖系统规则
2. **文档定位明确**：检索到的文档视为证据，而非可执行的命令
3. **工具调用强管控**：白名单 + 参数校验，高风险操作必须增加用户显式确认
4. **全链路可追踪**：输出和调用链路完整记录，出问题时可回放排查
5. **输入前置过滤**：输入清洗和风险分类，明显越权或诱导内容提前拦截

> 核心原则：不能靠一句 Prompt 防住所有注入，而是要靠系统设计让模型即使被诱导，也没有越权执行的能力。

## Q13. Agent 系统效果评估体系

**核心认知**

评估不能只看回答「像不像人」，要以**任务完成度**为核心。Agent 的分数不是单个模型的分数，而是整条链路的综合分数。

**四层评估维度**

| 层级 | 评估内容 |
|------|----------|
| 回答质量层 | 准确性、完整性、是否基于证据、是否存在幻觉 |
| 工具质量层 | 工具选择正确率、参数正确率、调用成功率 |
| 任务质量层 | 单次会话目标完成率、是否需要追问、是否需要人工接管 |
| 系统质量层 | 平均响应耗时、单次任务成本、Token 消耗、错误率和长期稳定性 |

**RAG 专项评估**（如系统包含 RAG 组件）

- 召回命中率
- 重排效果
- 答案引用质量（引用是否准确、来源是否标注）

## Q14. 线上 Agent 频繁选错工具的排查流程

按照从易到难、从静态到动态的顺序排查：

**第一步：检查工具定义**
- tool description 边界是否模糊
- 参数 schema 是否能有效区分不同场景
- 是否缺少禁用场景说明

**第二步：检查 Skill 路由层**
- 是否一次性返回了过多候选工具，造成噪声竞争
- 召回算法是否存在偏差

**第三步：检查 Prompt 体系**
- System Prompt 是否有错误引导
- Few-Shot 示例是否准确
- 最近是否修改过 Prompt

**第四步：回放全链路 Trace**
- 模型选工具前看到的完整上下文是什么
- 历史记忆是否污染了当前判断
- 用户输入是否存在严重歧义

> 排查核心：不要只看最后一步的选择结果，要搞清楚**候选工具集是怎么来的、模型为什么选了这个工具、执行前有没有做二次校验**。

## Q15. 避免 Agent 无限推理或死循环

**核心手段：状态机 + 硬限制**

**必须设置不可突破的硬阈值**

- 最大推理步数
- 最大工具调用次数
- 单任务最大 Token 开销
- 单任务最大执行时长

**明确状态定义**

提前定义清晰的状态转移规则：
- 什么状态代表任务完成
- 什么状态代表任务失败
- 什么情况需要请求用户补充信息

**重复动作检测**

出现以下情况，强制终止并返回阶段性结论：
- 连续多步调用同一个工具
- 连续得到相似的工具返回结果
- 工具调用没有带来任何新的证据或信息
