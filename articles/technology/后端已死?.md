# AI agent 根本不存在
我知道这话听起来扫兴。但你去拉勾、Boss 直聘上搜「AI Agent 工程师」，出来的要么是算法岗换了个名字，要么是培训机构的广告。

那 Agent 到底谁在做？两类人。

第一类是大厂的算法工程师。他们研究怎么让模型更好地 planning、怎么做 tool use、怎么搞 multi-agent 协作。这条路门槛很高——你得懂 LLM 原理、会炼丹、能读论文。说白了就是 AI 研究员加了个应用方向。

第二类是应用层的开发者。他们用现成的模型 API，搭各种 workflow。这活儿更像是「会调 API 的后端工程师」。懂点 prompt engineering，会写 Python，能把 LangChain 或者类似框架跑起来，就能开始干。

我自己属于第二类。白天写云架构，晚上折腾自己的 Agent。说几个实际的体会：

1、别被框架绑架。LangChain、AutoGPT、CrewAI 这些东西，学个思路就行。框架迭代太快，今天的最佳实践明天可能就是反面教材。核心能力是理解 Agent 的本质——感知、决策、执行的循环。

2、Prompt 工程比你想的重要。很多人觉得这玩意儿没技术含量。错了。一个好的 system prompt 能让 Agent 的可靠性提升一个量级。这不是玄学，是工程。

3、工具调用是关键。Agent 的能力上限取决于它能调用什么工具。API 集成、数据库操作、文件处理——这些传统后端技能，在 Agent 开发里全用得上。

4、容错和兜底机制。LLM 会犯错，会幻觉，会抽风。你得设计好失败后的处理逻辑。这块经验只能靠踩坑积累。

想入门的话，我的建议是：

先用 Claude 或 GPT 的 API 写一个能完成具体任务的脚本。不用框架，纯手撸。比如写一个能帮你总结邮件的、能帮你查天气发提醒的。从简单场景开始，感受「让 AI 替你干活」是怎么回事。

然后再去看那些框架。这时候你就知道它们解决的是什么问题，也知道哪些是过度设计。

最后说一句：现在这个领域变化太快，没有什么「成为 XXX 工程师」的标准路径。最靠谱的方式就是自己动手做东西，遇到问题解决问题。六个月后回头看，你会发现自己已经是了。
