## 一句话总结

LCM（Lossless Context Management）把 Agent 的记忆管理从"让模型自己写代码管"变成了"引擎提供结构化原语"，就像编程语言从 GOTO 进化到 for/while 一样。结果是：不仅在 1M token 级别任务上完胜 Claude Code，而且短对话零开销、历史消息永不丢失。

## 为什么这个问题重要？

任何一个用过 AI coding agent 的人都有过这种体验——聊着聊着，Agent 开始忘记前面讨论过的架构决策，或者重复问已经回答过的问题。这不是模型蠢，而是**上下文窗口塞爆了**。

问题有两个维度：

1. **硬限制**：即使是最新的模型有 1M+ 的窗口，多天的 agent 会话产生的工具调用、文件内容、中间推理加起来轻松超过这个上限。
2. **软退化**：在还没到硬限制之前，模型性能就已经开始下降，这就是所谓的"context rot"（上下文腐烂）。

传统的解决方案无非几种：滑动窗口（丢历史）、RAG 检索（碎片化）、让模型自己写脚本管理上下文。这些要么丢信息，要么靠模型自己即兴发挥，都不靠谱。

LCM 这篇论文给出的方案，是让**引擎**来管记忆，而不是让**模型**来管记忆。

## 核心思想：从 GOTO 到结构化控制流

论文作者打了一个非常精妙的比方：

> RLM（让模型自己管上下文）→ GOTO 语句（灵活但容易出错）
> LCM（引擎管上下文）→ 结构化控制流（for/while/if-else）（约束但可靠）

Dijkstra 在 1968 年提出"GOTO 有害论"之后，编程语言逐步用 for、while、if/else 替代了无限制的 GOTO。虽然理论上 GOTO 更灵活，能实现任何控制流，但实践证明了结构化原语更可靠、更容易推理、更不容易出 bug。

LCM 对 RLM（Recursive Language Models）的关系，完全就是这个逻辑的复刻。RLM 给模型完全的自主权——你自己写 Python 脚本去分块、去迭代、去压缩上下文。这非常灵活，但也继承了模型本身的不确定性：同一段分块逻辑，这次运行高效，下次可能就拉胯了。而且每次交互都要包一层递归脚手架，对于绝大多数短对话来说纯属浪费。

LCM 的答案是：**把控制流逻辑从随机层（模型）移到确定层（引擎）**。

## LCM 架构拆解

### 双态记忆：不可变存储 + 活跃上下文

LCM 把所有消息分成两层：

- **Immutable Store（不可变存储）**：每一句用户消息、每一次模型回复、每一个工具结果，都原样保存，永不修改。这是 Source of Truth。
- **Active Context（活跃上下文）**：真正发给模型看的窗口，由"最近的原始消息"和"旧消息的摘要节点"混合组装而成。

关键设计：摘要节点只是"物化视图"，是缓存，不是替代品。原始消息永远在那，随时可以通过 `lcm_expand` 工具无损还原。

### 分层摘要 DAG：多分辨率历史地图

LCM 的核心数据结构是一个**有向无环图（DAG）**，存储在支持事务写入和外键完整性的持久化数据库里（参考实现用嵌入式 PostgreSQL）。

当活跃上下文快满时，引擎会触发**递归上下文压缩**：旧消息被压缩成摘要节点，但摘要节点保留了指向原始消息的"无损指针"。多个摘要节点还可以进一步合并成更高层的"浓缩摘要"，形成一个多层级的 DAG。

这比传统方案强在哪？

- **flat file + grep**：只能精确匹配，搜不到"之前做了哪些架构决策"这种开放式问题。
- **RAG 语义搜索**：能处理开放式查询，但返回的是去语境化的碎片——谁说的、回应谁的、之后做了什么决定，这些上下文全丢了。
- **LCM DAG**：摘要节点提供了多分辨率的"历史地图"，同时底层的无损指针允许定点钻取原始内容。完美结合。

### 三级升级协议：保证收敛

一个已知的坑是"压缩失败"——模型被要求摘要一段文字，结果输出比输入还长。RLM 这类依赖模型自己写控制流的方案必须处理这种情况。

LCM 的解决方案非常务实：**三级升级协议**。

1. **Normal（正常）**：LLM 摘要，保留细节。
2. **Aggressive（激进）**：如果一级没减少 token 数，升到要点摘要。
3. **Deterministic Fallback**：如果还不行，直接确定性截断到 512 token——完全不需要 LLM 参与，保证收敛。

### 零开销延续：短对话不付出代价

这是 LCM 非常务实的一个设计。很多递归架构有个问题：即使只有两句话的简单对话，也要初始化 REPL、加载 prompt、解释执行代码，"杀鸡用牛刀"。

LCM 的分层开销非常清晰：

- **低于软阈值**（大多数对话）：零开销，存储只做被动日志，用户体验就是裸模型的延迟。
- **软阈值和硬阈值之间**：异步压缩，不阻塞用户。压缩完成后原子性替换上下文——用户几乎感知不到。
- **超过硬阈值**：阻塞压缩，但实践中很少触发。

### 作用域缩减不变量：防止无限委派

Agent 系统里另一个常见的坑是无限委派：Agent 把自己的全部任务委派给子 Agent，子 Agent 又委派下去，形成永无止境的链条。

LCM 的解决方案叫**作用域缩减不变量**：当子 Agent 想再 spawn 子 Agent 时，必须声明两个参数——委派了什么工作（delegated_scope），自己保留了什什么工作（kept_work）。如果它说不清自己留了什么（即全委派出去了），引擎直接拒绝调用，让它自己干。

这个设计确保每层委派都必须是严格的作用域缩减，从根本上保证递归必须触底。**不需要设定硬性的深度上限。**

## Operator-Level Recursion：让模型告别手写循环

这也是 LCM 最漂亮的设计之一。

RLM 的"符号递归"让模型自己写 `for chunk in context: ...` 来管理上下文。LCM 取而代之的是两个引擎原语：

- **LLM-Map**：对列表中的每一项发起独立的 LLM 调用，并行执行。适合分类、提取、评分等无副作用的纯函数任务。
- **Agentic-Map**：对每一项 spawn 一个完整的子 Agent 会话，子 Agent 有完整的工具访问权限。适合需要多步推理或与环境交互的任务。

几个关键设计细节：

- **并发管理**：引擎用悲观锁管理 worker pool，保证 exactly-once 语义，失败自动重试。
- **上下文隔离**：输入输出都在磁盘 JSONL 文件里，完全在活跃上下文之外。模型可以处理任意大小的数据集，而不污染上下文窗口。
- **Schema 验证输出**：每次调用都附带 JSON Schema，引擎验证每个输出，不合格就自动注入错误提示让模型重试。下游聚合代码可以信赖输出结构的一致性。
- **递归组合**：Agentic-Map 的子 Agent 本身也可以调用 Agentic-Map，形成递归并行。例如：顶层按仓库 map，每个仓库内部按文件 map。

## 实验结果：真的比 Claude Code 强

评测用的是 OOLONG 基准测试，模型都是 Opus 4.6（保证公平）。

![OOLONG 基准测试性能对比](https://wynneyin.oss-cn-hangzhou.aliyuncs.com/arxiv/2026-05-08-fig2-oolong-performance-comparison.png)

*OOLONG 基准测试上的性能对比。LCM 在超长上下文场景下优势尤为明显。*

几个亮点数据：

- **整体**：Volt（LCM）74.8 分 vs Claude Code 70.3 分，领先 4.5 分。
- **短上下文**（<32K）：二者基本持平，LCM 不拖后腿也不占便宜。
- **256K**：Volt 领先 10 分（+18.5 vs +8.5）。
- **512K**：Volt 领先 12.6 分（+42.4 vs +29.8）。
- **1M**：Volt 仍然领先 4.3 分（+51.3 vs +47.0）。

为什么在超长上下文里优势这么明显？因为 Claude Code 依赖模型自己去分块、去追踪状态，在百万级 token 面前，模型既要管推理又要管记忆管理，顾此失彼。而 LCM 把数据处理完全移到引擎层并行执行，模型根本不用看原始数据集——只需要指定每条该干嘛。

### 数据污染的小插曲

论文还提了一个有趣的发现：Opus 4.6 有时直接认出了 OOLONG 里的原始数据集，从训练记忆里直接出答案，跳过了聚合推理。比如某次 Claude Code 的推理过程直接写"我对照了 TREC QC 数据集，3182 个问题的标签完全匹配……"

作者的处理方式是：排除所有推理轨迹中显示模型"认出数据集"的任务，只报告去污后的结果。去污后 Volt 依然领先，差距反而更大（因为 LCM 的 Map 架构本身就更鼓励逐条推理而非走捷径）。

## 局限和我的看法

**局限**：

1. **评测基准的脆弱性**：OOLONG 是静态数据集，迟早会被所有模型"背下来"。论文自己也承认需要动态生成的评测基准。
2. **只是 map 没有 reduce**：论文实现了 Map 侧，但 Reduce 侧靠模型自己写代码，没有提供 LLM-Reduce 原语。作者说"reduce 用确定性代码更好"，但这可能限制了某些场景。
3. **依赖特定模型组合**：测试用了 Opus 4.6 + Haiku 4.5 的组合，不同模型下的表现还需要验证。

**我的看法**：

这篇论文最打动我的不是它打了个 Claude Code——那个更像是营销卖点。真正厉害的是**架构思想的清晰度**。

GOTO → 结构化控制流的那个类比真的妙。一个领域的成熟，往往就体现在"把最灵活的方案约束成最可靠的方案"这个过程里。AI Agent 现在还处在"大家各显神通"的早期阶段，LCM 提供了一个"结构化 Agent 记忆管理"的范本。

具体到我自己用 OpenClaw 的场景：OpenClaw 本身就面临类似的问题——长时间会话后上下文膨胀，需要压缩。LCM 的几个设计思路（DAG 摘要、三级升级、零开销延续、作用域缩减）都值得参考。

尤其是"把记忆管理从模型层提到引擎层"这个思路——让模型专注推理，让引擎负责记忆。这种分工，我觉得会是 Agent 架构的未来方向。
